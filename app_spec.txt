<!--
  claw-forge-plugin — Project Specification
  ==========================================
  Generated from docs/PRD.md and docs/ARCHITECTURE.md.

  This is the build spec for the Claude Code plugin that pairs with the
  claw-forge sidecar (Python package). The plugin executes inside the
  user's interactive Claude Code session so agent invocations bill against
  their Pro/Max subscription pool instead of the metered Agent SDK
  allowance.

  Hard invariant: no `claude_agent_sdk` imports anywhere in this tree.
  All agent work happens via Task subagents, slash commands, and skills
  loaded into the host session.

  Note: <database_schema> and <api_endpoints_summary> are intentionally
  omitted — the plugin owns no schema and defines no API. State persists
  in the sidecar's existing .claw-forge/state.db (accessed only through
  the sidecar CLI) and the sidecar CLI contract surface is documented in
  docs/PRD.md §8 and <claw-forge>/docs/sidecar-contract.md.
-->
<project_specification mode="greenfield">
  <project_name>claw-forge-plugin</project_name>

  <overview>
    A Claude Code plugin that surfaces the claw-forge spec → DAG → parallel
    autonomous agents workflow as slash commands, skills, and Task-tool
    subagents — billed against the user's Pro/Max subscription pool. It
    replaces the Agent-SDK-based dispatch path in sidecar v0.5.x while
    keeping the sidecar's state service, Kanban UI, spec validator, and
    git/worktree machinery untouched. Targets plugin v0.1.0, paired with
    sidecar claw-forge v0.6.0.
  </overview>

  <target_audience>
    Existing claw-forge users on Claude Pro or Max plans who want to keep
    using claw-forge without metered API spend on top of their
    subscription. Also greenfield/brownfield authors who already work in
    Claude Code daily and want spec-driven autonomous agents inside their
    normal workflow.
  </target_audience>

  <technology_stack>
    <frontend>
      <framework>None — the plugin has no UI. The Kanban UI ships with the sidecar at localhost:8421.</framework>
    </frontend>
    <backend>
      <framework>Markdown — each slash command, skill, and subagent is a markdown file loaded by the host Claude Code session.</framework>
      <shell>POSIX sh for hooks (SessionStart bootstrap)</shell>
      <manifest>Claude Code plugin manifest (.claude-plugin/plugin.json)</manifest>
    </backend>
    <database>
      <primary>None directly. State persistence is owned by the sidecar's existing SQLite store at .claw-forge/state.db, accessed exclusively through the sidecar CLI — never read or written by the plugin.</primary>
    </database>
    <communication>
      <api_style>Bash + sidecar CLI (every JSON-emitting subcommand listed in PRD §8 is the contract surface).</api_style>
      <response_format>JSON envelopes emitted by sidecar CLI subcommands with --json flag, documented in &lt;claw-forge&gt;/docs/sidecar-contract.md.</response_format>
    </communication>
    <infrastructure>
      <distribution>Plugin marketplace install via /plugin install claw-forge (or direct repo URL for v0.1)</distribution>
      <ci>GitHub Actions: markdownlint + shellcheck + JSON-schema validation + no-SDK invariant grep + cross-repo CLI smoke test</ci>
    </infrastructure>
  </technology_stack>

  <prerequisites>
    - Claude Code CLI installed and authenticated to a Pro or Max subscription
    - Sidecar claw-forge Python package on $PATH at version >= 0.6.0 (pip install -U claw-forge)
    - Git 2.30+ for worktree management (used by sidecar git helpers)
    - POSIX shell, jq, awk available for the SessionStart hook
    - For CI: Node.js 18+ (markdownlint), shellcheck, jq, GitHub Actions runner
  </prerequisites>

  <key_interactions>
    <interaction number="1" name="Fresh install &amp; first greenfield run">
      1. User runs `pip install claw-forge` (v0.6.0+)
      2. User runs `/plugin install claw-forge` inside Claude Code
      3. User cd's into a project dir, creates claw-forge.yaml, restarts the session
      4. SessionStart hook detects the project, version-gates the sidecar, boots state service, prints banner
      5. User runs `/claw-forge spec-create` → spec-authoring skill loads → produces app_spec.xml
      6. User runs `/claw-forge plan` → sidecar parses spec, seeds DAG
      7. User runs `/claw-forge run` → dispatch-loop skill loads → wave-by-wave Task calls until DAG complete
      8. User runs `/claw-forge merge` → squash-merges all ready feature branches to target
    </interaction>
    <interaction number="2" name="Single-task bugfix on failure">
      1. /claw-forge run completes with one failed task
      2. User runs `/claw-forge fix &lt;task-id&gt;`
      3. bugfix-loop skill loads, fetches task record + HANDOFF.md + prior commits via sidecar CLI
      4. Dispatches bugfix-task subagent into the existing worktree with structured resume context
      5. Subagent fixes the issue, commits, returns structured summary
      6. /claw-forge merge picks up the now-passing branch
    </interaction>
    <interaction number="3" name="Migration from sidecar v0.5.x">
      1. User runs `pip install -U claw-forge` (brings v0.6.0)
      2. Existing claw-forge.yaml: providers: block now ignored (no error, no removal required)
      3. agent.isolation: container setting now ignored (one-line warning at sidecar startup if set)
      4. User runs `/plugin install claw-forge` inside Claude Code
      5. Habit change: `claw-forge run` → `/claw-forge run`. Same for fix, merge, boundaries apply.
      6. Existing .claw-forge/state.db continues to work unchanged
    </interaction>
  </key_interactions>
